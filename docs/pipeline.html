The shape of a run

A run drives one concept (your topic) through one persona toward a published page. Work happens in two phases — research, then a build → check → refine loop — guarded by four ordered gates. Three more things sit around the gates:

Guardrails (Pillar 2)

Shape and police a draft against the persona. Two halves: a preventive compile (persona → system prompt) and detective validators (cheap deterministic checks on the built page).

Checkpoints / Gates (Pillar 3)

Four ordered pass/fail gates with explicit scores and thresholds: research‑sufficiency, voice‑fidelity, design‑conformance, quality. Each returns autoCorrectable, which decides loop vs. escalate.

The agent (Anthropic, via the Vercel AI SDK) only ever sees a compiled system string and a prompt. Everything below is the machinery that builds that string, judges the result, and turns the winning draft into a hostable file.

A guardrail has two halves

The Guardrails pillar is split so the same persona both shapes the first attempt and polices the result — without an LLM in the policing path.

Preventive deterministic

compilePersonaSystem(persona) flattens the persona into the Anthropic system prompt — voice, exemplar, style, learnings, design tokens. Same persona → byte‑identical output; never throws.

Detective deterministic

buildValidators(persona) returns pure (page, persona) → findings functions — design‑token presence, banned‑phrase scan, structure. These power the design‑conformance gate. No network, no model.

The run loop, step by step

This is the exact control flow in run-engine.ts. Caps: MAX_RESEARCH_ATTEMPTS = 2 MAX_ATTEMPTS = 2 (initial + one retry each).

1
Compile the persona

Build system + detective validators from the persona. Seed the run context (concept, attempt = 1).

2
Research

agent.research({ system, concept }) → research text + sources.

G1
Gate · Research sufficiency deterministic
pass (≥ 3 sources) → proceed to build.
fail → re‑research once with the gate’s guidance; still failing → escalate (“topic is research‑light”).
3
Build (or refine)

agent.build({ system, research, feedback? }) → a Webpage object. Attempt 1 is a build; later attempts are refine and carry the composed feedback.

G2–4
Gates · voice → design → quality, in order

Each emits a draft event with its score so the live UI shows the verdict per attempt.

all pass → pause at the human approval gate (does not auto‑publish).
hard fail (any gate with autoCorrectable:false) → escalate immediately.
soft fail → if attempts remain, compose feedback & refine (back to step 3); else escalate (“exhausted attempts”).
4
Human approval → Publish

On approve_anyway, the last draft is published: rendered to a self‑contained HTML file + a receipt. See Output → web page.

The four gates in detail

Every gate returns the same shape: { passed, score, threshold, details, autoCorrectable, feedback?, alarms[] }. autoCorrectable is the load‑bearing flag — true ⇒ the loop may refine; false ⇒ escalate to a human.

Gate 1 · Research sufficiency deterministic

threshold: ≥ 3 distinct sources autoCorrectable: false alarm: INSUFFICIENT_RESEARCH

Runs before any build. Counts distinct, non‑empty entries in research.sources (trimmed + de‑duped). No LLM — the demo can rely on identical pass/fail every time.

score = distinctSources.length  ·  pass = score >= RESEARCH_MIN_SOURCES (3)

You can’t “refine” your way to more sources, so failure is not auto‑correctable: the engine re‑researches once with the gate’s feedback; if it still can’t clear the bar it escalates with “Topic appears research‑light: only N source(s)… need ≥ 3.”

Reality check: the default AI‑SDK AnthropicAgent.research() has no server‑side web search yet, so it returns sources: [] → this gate fails → re‑research → escalation. Only the anthropic-research worker (raw SDK + real web_search) populates sources.

Gate 2 · Voice fidelity judge

threshold: VOICE_THRESHOLD = 0.75 autoCorrectable: true alarm: VOICE_DRIFT on judge error: CHECKPOINT_ERROR

Scores the built page against the persona’s voiceSample + style. The judge is injectable: a deterministic offline judge by default, swappable for a real LLM judge via deps. Pass iff score ≥ 0.75.

passed = judgeScore >= 0.75   →   on fail, feedback quotes the voiceSample + stylePoints to redraft against

Fail‑closed: if the judge throws, the gate returns score 0, autoCorrectable:false, and a critical CHECKPOINT_ERROR — a broken judge can never pass a page (never fail‑open).

backend/src/checkpoints/voice-fidelity.ts

Gate 3 · Design conformance deterministic

threshold: 1.0 — ALL findings must pass autoCorrectable: true (on failures) alarm: DESIGN_DRIFT

Runs the persona’s detective validators over the built page and folds their findings into a score. Pass iff every finding passes.

score = (total − failed) / total  ·  pass = (failed.length === 0)

If there’s no page yet or no validators wired, it passes vacuously (score 1, autoCorrectable:false). Three validator families run:

ValidatorChecksRule id
designTokenDeclared typography / layout / palette values appear as marker fragments in the page’s CSS+HTML. tone is a soft signal (not hard‑gated). Uses antiMarkers so e.g. serif doesn’t match inside sans-serif.design-token:<key>
bannedPhrasePage text must not contain character‑leak phrasings (see floor below); persona may extend via a bannedPhrases token. One finding per hit.banned-phrase
structureNon‑empty title; at least one <h1>; body text ≥ 200 chars.structure:title · :heading · :length

Gate 4 · Quality judge

threshold: QUALITY_THRESHOLD = 0.75 autoCorrectable: true alarm: INSUFFICIENT_QUALITY on judge error: CHECKPOINT_ERROR

The final gate: is the page good enough to publish? Same injectable‑judge + fail‑closed contract as voice fidelity. Pass iff score ≥ 0.75; on fail it asks for more depth and a complete summary.

backend/src/checkpoints/quality.ts

The judges (default = deterministic, LLM‑swappable)

Both judge gates take an injectable Judge: (input) → score in [0,1]. The defaults are token‑free heuristics so the whole harness develops and tests offline; passing a different Judge swaps in a real model behind the same signature. runJudge() wraps every judge fail‑closed — a throw becomes { ok:false, score:0 }.

Deterministic voice judge

How much of the persona’s voiceSample vocabulary the page echoes (Jaccard‑style), plus a small style bonus, minus a penalty for hype words the personas avoid.

coverage = sampleCoverage·0.85 + styleBonus·0.15 penalty = min(0.6, hypeHits · 0.15) score = clamp01(coverage − penalty)

Hype markers: synergy, leverage, disruptive, paradigm, stakeholder, roi, verticals, maximize.

Deterministic quality judge

Two cheap proxies for “did the worker produce something real”: body length and a non‑trivial summary.

contentScore = clamp01(bodyWords / 60) summaryScore = clamp01(summaryWords / 8) score = clamp01(contentScore·0.7 + summaryScore·0.3)

Refine, escalate & the human approval gate

What the loop does after a round of gates is decided entirely by the gate results:

  • all passPause at the human approval gate (HITL). Status → awaiting_approval. The page is not auto‑published.
  • hard failAny gate with autoCorrectable:false (e.g. a judge error, or research‑light) → escalate immediately.
  • soft failIf attempt < MAX_ATTEMPTS → compose feedback & refine. Otherwise escalate (“exhausted N attempts”).

Refine feedback is composed by the checkpoints, not the engine

nextBuildFeedback(results) concatenates every failed gate’s feedback, each tagged with its gate name, into the feedback string the next build receives:

- [voice-fidelity] The draft reads off-voice for <persona> (scored 0.61, need 0.75)…
- [design-conformance] Fix these design rules: design-token:typography (…)

Escalation options the human is offered

enrich_persona approve_anyway retry abort

enrich_persona reloads the edited persona and recompiles system + validators, then re‑enters from a fresh attempt — so a tweak to voice or design takes effect immediately. approve_anyway publishes the last draft as‑is.

Turning the output into a web page

The build phase doesn’t emit raw text — it returns a typed object via the AI SDK’s generateObject, validated against a Zod schema. That object is what gets rendered and hosted.

1 · The typed output contract

WebpageSchema = z.object({
  title:       z.string().min(1),   // required
  html:        z.string().min(1),   // required — the body markup
  css:         z.string(),          // inlined into <style>
  summary:     z.string(),          // metadata (judged, not rendered)
  sourcesUsed: z.array(z.string()), // metadata / provenance
})

The agent is forced to produce exactly this shape (generateObject({ schema: WebpageSchema, system, prompt })), so the worker can never return prose the sink can’t publish.

2 · Render to a single self‑contained file

renderHtml(page) assembles one document — the title is HTML‑escaped, the css is inlined into a <style> tag, and the html body is placed verbatim. summary and sourcesUsed stay as metadata and are not written into the file.

<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1" />
  <title>{escaped title}</title>
  <style>{css}</style>
</head>
<body>
  {html}
</body>
</html>

3 · The “truly standalone” property

isSelfContained(html) is the Pillar‑1 guarantee — the file must render with zero network. Inline <style>/<script> are fine; it rejects any of:

<link rel="stylesheet">   ·   <script src="…">   ·   @import

4 · Publish & receipt

On approval, createFileSink.publish() writes <runId>.html to the published directory and returns a receipt; the engine stores the final webpage, emits a published event over the live stream, and flips the run to published.

Receipt = {
  id,                                  // runId
  url:        `${baseUrl}/published/${runId}`,
  bytes,                               // byte length of the file
  publishedAt,                         // ISO timestamp
  workerId,                            // which worker produced it
}

The file is then served at GET /published/:id, and the published SSE event is what surfaces the finished page in the live run view and the Gallery.
