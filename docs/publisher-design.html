01The Challenge

Build a harness — the framework an AI agent lives inside. The harness, not the agent, is what's evaluated: guardrails that constrain behavior, checkpoints that evaluate outputs, clean material handling in and out, and alarms that fire when something goes wrong.

Must

  • All four pillars implemented and demonstrably separate from the worker.
  • Agent behavior changes meaningfully from guardrail/checkpoint feedback.
  • Guardrails are declared, not implicit. Checkpoints have explicit pass/fail.
  • Alarms produce structured output — named type, context, severity, recommended action.
  • Runs on a real input from your own work at demo time.
  • An HARNESS.md covering architecture and design.

Should & Bonus

  • Swappable agent — drop in a different agent with no harness changes.
  • Persisted checkpoints — replay a run from any checkpoint forward.
  • Human-in-the-loop escalation — the harness knows when to stop and ask.
  • Bonus A second worker swapped in during the demo to prove portability.

Deliverables: 1-page planning doc (Fri 11:30 PM) · repo + deployed URL + HARNESS.md + 5-min demo (Sat 4:30 PM)

02Product Vision

Publisher lets anyone turn a research concept, idea, or thought experiment into a unique, elegantly-designed single-page webpage — written in the authentic voice of their persona. The agent researches and writes; four separate pillars guarantee the output captures the persona's voice and design, is backed by sufficient research, and meets a quality bar — or it refuses to publish and says exactly why.

The persona is the heart. In a brief onboarding, a user defines a persona — its voice, key style points, key learnings, and design elements. It's saved and reused: it is simultaneously the declared guardrail and the context handed to the agent. One user can keep several personas and publish in each.

This maps cleanly to the challenge thesis — "Agents focus on tasks. Harnesses focus on constraints. A well-designed harness makes constraint-handling invisible to the agent." The agent just writes; the cage makes it sound like you, look like you, and be worth reading.

03The Four Pillars, Mapped

PillarIn PublisherExample signal
MATERIAL In: concept/idea + selected persona. Out: a self-contained single-page webpage (static, hosts anywhere). Typed contract: {title, html, css, summary, sourcesUsed}. Source & sink adapters; clean in/out interface.
GUARDRAILS The persona is the declared guardrail — voice rules, style points, design tokens. Compiled preventively into the system prompt and detectively into validators. Declared once, enforced twice. Output sounds & looks like that persona.
CHECKPOINTS Ordered pass/fail gates: research-sufficiency · voice-fidelity · design-conformance · quality. Persisted to a journal → replay from any checkpoint. A failed gate is a real outcome: "couldn't publish — insufficient research."
OBSERVABILITY & ALARMS Meters token cost per phase, latency, error rate; emits structured alarms on budget breach or gate failure — each with type, severity, context, recommended action. TOKEN_BUDGET_EXCEEDED, VOICE_DRIFT, INSUFFICIENT_RESEARCH, WEBPAGE_GENERATION_FAILED.

Observability is the metric layer; alarms are the structured pillar output that fires when a metric breaches a declared budget or a gate fails.

04Architecture

Eight components. The four graded pillars are highlighted. The agent is deliberately dumb about all of them — it researches and writes; the persona and the pillars do the constraining.

Orchestrator
Thin run-loop and the only thing the agent talks to. Owns lifecycle, retry loop, checkpoint persistence, replay. No domain logic.
Material Handling
Source brings a concept + persona in; Sink emits a static single-page webpage out.
Guardrails (Persona)
Declared persona compiled into a preventive prompt fragment + detective validators.
Checkpoints
Ordered pass/fail gates with thresholds, persisted to the run journal.
Observability & Alarms
Telemetry → structured, named, severity-tagged alarms.
Persona Store
Per-user personas (voice, style, learnings, design) in SQLite, created in onboarding.
Escalation / HITL
Critical alarm or non-correctable failure → pause and ask the operator in the UI.
Agent (swappable)
In-process module behind one seam — Vercel AI SDK over @ai-sdk/anthropic. Research → build → refine.

05The Control Loop — where the agent's behavior changes

1. persona   = PersonaStore.load(personaId)        // declared voice + style + design
2. material  = Material.load(concept, persona)
3. guardrails = compile(persona)                   // prompt fragment + validators

   ── RESEARCH ──
4. research = await agent.research(persona, concept)        // web_search / web_fetch
5. if research-sufficiency FAILS:
       INSUFFICIENT_RESEARCH → append "go deeper on X" → research again   ◀ BEHAVIOR CHANGES

   ── BUILD ──
6. webpage = await agent.build(persona, research)          // typed {title, html, css, …}
7. for cp in [voice-fidelity, design-conformance, quality]:
       result = cp.evaluate(webpage, persona, material)
       journal.persist(result)                              // ◀ replay point
       if result.FAILED:
           if autoCorrectable AND attempts < MAX:
               append feedback(result) → build again        ◀ BEHAVIOR CHANGES (refine)
           else:
               raise alarm; if CRITICAL → escalate()        // stop and ask the user

8. receipt = Sink.publish(webpage)   // only persona-faithful, researched, quality pages ship
The meaningful behavior change: a failed VOICE_DRIFT checkpoint ("too formal, 0.42 < 0.75") produces structured feedback fed back into the next build prompt ("match this voice sample"). The redraft scores 0.81 and passes. Guardrails (the persona) shape the first attempt; checkpoint feedback corrects the later ones. That closed loop is the harness.

06Agent Integration

The wrapper is the Vercel AI SDK (ai + @ai-sdk/anthropic). Swapping the worker — a different Claude model, or a different provider — is a one-line model change, which is the portability bonus.

NeedAI SDK
System promptsystem: (the persona); cache via providerOptions.anthropic.cacheControl
Multi-turn contextWe hold the ModelMessage[] → maps to the SQLite journal & replay
Tools + looptool({inputSchema, execute}) with stopWhen: stepCountIs(n)
Researchanthropic.tools.webSearch_20250305() + webFetch_20250910() (provider-executed)
Material-outgenerateObject({ schema }) → typed {title, html, css, summary, sourcesUsed}
StreamingstreamText() → UI over WebSocket
Telemetryresult.usage + providerMetadata.anthropic (cache tokens) → Observability
Honest portability tradeoff: the AI SDK makes the core loop provider-portable, but the two pieces that buy the most — server-side web search and prompt caching — are Anthropic-specific (reached via providerOptions/provider tools). So research depth won't carry to OpenAI/Google as-is; the build/refine loop will. The precise answer to "how portable?" is "core loop portable; research tools provider-scoped" — not "fully portable."

Models: claude-opus-4-8 for research/build; claude-haiku-4-5/claude-sonnet-4-6 for the cheaper LLM-judge checkpoints.

07Key Decisions & Why

DecisionWhy, over the alternative
The persona is a declared guardrail, not a hidden promptVoice/style/design are inspectable, reusable data authored in onboarding; burying them in a prompt makes them implicit and unenforceable. The rubric demands declared guardrails.
Agent never talks to the pillars; the orchestrator mediatesThis is the thesis — constraint-handling stays invisible to the worker. Letting the agent call pillars itself leaks the cage and breaks swappability.
Guardrails enforced twice (prompt + validator)Prompt-only is unreliable; validator-only wastes a full generation before catching drift. Defense-in-depth.
Checkpoints separate from guardrailsPersona = declared rule; checkpoint = measured gate. Checkpoints also cover quality gates that aren't behavioral rules (research-sufficiency, coherence).
Observability feeds Alarms; alarms are structured, not thrownAlarms need severity + recommended action + context, and warnings must not halt. Exceptions lose that and the per-phase cost story.
Deterministic gates hard-block; LLM-judged gates escalateSecret/research-count/budget are deterministic hard blocks; voice/design/quality use a judge with a threshold and prefer escalation over silent block.
Vercel AI SDK over the raw Anthropic SDKProvider abstraction → one-line worker swap = the portability bonus. Tradeoff stated: web tools + caching are Anthropic-specific.
Harness owns the loop (no autonomous framework)The harness must own the loop so a checkpoint runs after every round. Agent-SDK/Managed-Agents self-drive, which we don't want.
SQLite for critical recordsRun state (personas↔runs↔checkpoints↔alarms↔metrics) is relational and queried live by the API. Published webpages stay static files.

08Known Tradeoffs & Uncertainties

Stated out loud — the architecture review rewards honesty about gaps.

  1. Fuzzy checkpoints rely on LLM-as-judge (voice, quality) — non-deterministic, can mis-score. Deterministic gates hard-block; fuzzy gates threshold + escalate. Calibration is genuinely uncertain.
  2. Defining a persona's voice well enough to measure is hard; a thin persona is a weak guardrail. Repeated VOICE_DRIFT → escalate to enrich it.
  3. Research-sufficiency is a judgment call — a declared, per-persona-tunable threshold, not a hard constant.
  4. The retry loop can oscillate — bounded by MAX_ATTEMPTS; on exhaustion → WEBPAGE_GENERATION_FAILED + escalate.
  5. Third-party model dependency — refusal/rate-limit/outage stalls a run. Structured alarms + retry/escalate; swappable seam = second provider is an adapter.
  6. Cost & latency on the critical path — prompt-cache the persona+research prefix; meter per-call usage; alarm before spend runs away.
  7. Portability is partial — core loop portable; research tools + caching provider-scoped (see §06).
  8. Scope risk for 24h — protect the demo spine; everything else is additive.

11Defense Q&A — the sharpest shots

Q · Isn't "the persona" just a fancy system prompt?
No — it's declared, inspectable, reusable data authored in onboarding, compiled into both a preventive prompt fragment and detective validators, and measured against by the voice/design checkpoints. A system prompt is implicit and unenforced.

Q · Your 4th pillar is "observability," but the rubric says "alarms"?
Observability is the metric layer (token cost/phase, latency, error rate); alarms are the structured pillar output — named type + severity + context + recommended action — that fires when a metric breaches a budget or a gate fails.

Q · How does the agent's behavior actually change?
A failed checkpoint produces structured feedback injected into the next build. VOICE_DRIFT 0.42 → "match this sample" → the refined page scores 0.81 and passes. Live in the demo.

Q · How portable is the swap, really?
The Vercel AI SDK makes the core build/refine loop provider-portable (one-line model change). Server-side web search and prompt caching are Anthropic-specific — research depth is provider-scoped. Honest, and stronger than claiming "fully portable."

Q · How do you measure "voice fidelity"? That's subjective.
An LLM judge scores the page against the persona's voice profile + sample against a declared threshold — genuinely uncertain, so it's advisory-with-escalation while deterministic gates (research count, token budget) are the hard blocks.

Q · What if the model refuses, rate-limits, or the provider is down?
Check finishReason and catch SDK errors: REFUSAL/PROVIDER_ERROR escalate; RATE_LIMITED backs off and retries. Every run resumes from its last checkpoint.

Q · Isn't this just a generic agent orchestrator?
No — the product is the four constraint pillars for persona-faithful publishing. A generic orchestrator routes tasks; Publisher governs a research-and-build worker and refuses to ship anything off-voice, under-researched, or low-quality.
